## Design Context

### Users
- Primary users are super admins, data operations managers, and internal platform operators working in a dataset annotation platform.
- They use the product in a high-frequency operational context to manage datasets, monitor pipeline progress, configure automated labeling, manage users and teams, and make downstream routing decisions.
- The interface should help them scan status quickly, understand workflow progression, and act confidently without visual noise.

### Brand Personality
- Calm
- Operational
- Polished

The product should feel stable, clear, and professional rather than flashy or experimental. It should communicate control, reliability, and execution clarity.

### Aesthetic Direction
- Light-first enterprise admin interface
- English-only UI
- Structured and deliberate page composition
- White or near-white surfaces on a soft neutral background
- Primary color: #8B5CF6
- Secondary color: #D946EF
- Brand gradient reserved for key CTAs and small emphasis moments
- Typography uses Poppins for interface text and JetBrains Mono for technical values

Reference direction:
- Calm and operational, but visually refined
- Strong spacing rhythm and clear hierarchy
- Workflow-heavy screens should feel like process control surfaces, not generic component demos

Anti-direction:
- No generic AI dashboard aesthetics
- No heavy card nesting
- No brand blue as primary visual color
- No decorative excess, glassmorphism, or loud gradients

### Design Principles
1. Make workflow state legible first. Users should understand progress, current stage, and next action at a glance.
2. Use composition over decoration. Hierarchy should come from spacing, grouping, alignment, and type before borders or effects.
3. Keep brand accents rare and intentional. Purple and magenta should guide attention, not fill the interface.
4. Prefer large calm surfaces and a few strong groups over many small boxes.
5. Every screen should feel operational, polished, and consistent with the same product family.
6. Prefer extending shared primitives over page-local styling. Reuse shadcn/ui and shared layout shells before introducing one-off wrappers.
7. Keep custom CSS near zero. If a pattern repeats, extract it into a reusable primitive instead of copying class strings across pages.
8. Product-facing mock data should default to a North American business context for names, companies, and phone formats unless another locale is explicitly requested.
9. Tables must remain width-constrained to the viewport. When a table is wider than the available content area, horizontal scrolling must happen inside the table container, never by expanding the full page.
